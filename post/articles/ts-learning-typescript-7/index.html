TypeScript 러닝타입스크립트

[TIL] 러닝타입스크립트 - 7. 인터페이스

Posted by Hyewon Kang on Sun, Apr 30, 2023

⚠️ 러닝 타입스크립트 책의 모든 내용을 담지 않으며 주관적인 기준으로 필요한 내용만 작성하고, 부족한 내용에 대해 추가 개념을 정리합니다.

타입 별칭 vs. 인터페이스

  • 인터페이스는 일반적으로 더 읽기 쉬운 오류 메시지, 더 빠른 컴파일러 성능, 클래스와의 더 나은 상호 운용성을 위해 선호된다.

💡 NOTES

  • 인터페이스는 속성 증가를 위해 병합(merge)할 수 있다. 이 기능은 내장된 전역 인터페이스 또는 npm 패키지와 같은 외부 코드를 사용할 때 특히 유용하다.
  • 인터페이스는 클래스가 선언된 구조의 타입을 확인하는 데 사용할 수 있지만, 타입 별칭은 사용할 수 없다.
  • 일반적으로 인터페이스에서 타입스크립트 타입 검사기가 더 빨리 작동한다. 인터페이스는 타입 별칭이 하는 것처럼 새로운 객체 리터럴의 동적인 복사 붙여넣기보다 내부적으로 더 쉽게 캐시할 수 있는 명명된 타입을 선언한다.
  • 인터페이스는 이름 없는 객체 리터럴의 별칭이 아닌 이름 있는(명명된) 객체로 간주되므로 어려운 특이 케이스에서 나타나는 오류 메시지를 좀 더 쉽게 읽을 수 있다.

속성 타입

  • 읽기 전용 속성 (readonly 제한자) → interface에서만 사용 가능

    1interface Page {
2    readonly text: string; // text 속성을 읽을 수 있지만 새로운 값으로 재할당 불가
3}

    • readonly 인터페이스 멤버는 코드 영역에서 객체를 의도치 않게 수정하는 것을 막는 편리한 방법이다. → 단지 타입 검사기를 사용해 개발 중에 그 속성이 수정되지 못하도록 보호하는 역할을 한다.

       1interface Page {
 2    readonly text: string;
 3}
 4
 5function read(page: Page) {
 6    console.log(page);
 7    // page.text += '!'; // ERROR
 8}
 9
10const pageIsh = {
11    text: 'Hello, world!',
12};
13
14pageIsh.text += '!';
15
16read(pageIsh);

  • 호출 시그니처 (call signature)

    • 호출 시그니처는 값을 함수처럼 호출하는 방식에 대한 타입시스템의 설명이다. → 타입스크립트에서 함수의 타입을 지정할 때 사용하는 문법
    • 호출 시그니처가 선언한 방식으로 호출되는 값만 인터페이스에 할당할 수 있다.

  • 인덱스 시그니처 (index signature)

     1interface DatesByName {
 2    [i: string]: Date;
 3}
 4
 5const publishDates: DatesByName = {
 6    Frankenstein: new Date('1 January 1818'),
 7};
 8
 9publishDates.Frankenstein; // 타입: Date
10console.log(publishDates.Frankenstein.toString()); // OK
11
12**publishDates.Beloved;** // 타입은 Date 이지만 런타임 값은 undefined
13**console.log(publishDates.Beloved.toString()); // TypeError**: Cannot read properties of undefined (reading 'toString')

    • 인덱스 시그니처는 객체에 값을 할당할 때 편리하지만 타입 안정성을 완벽하게 보장하지는 않는다.

      ⇒ 키/값 쌍을 저장하려고 하는데 키를 미리 알 수 없다면 Map을 사용하는 편이 더 안전하다.

    • 숫자 인덱스 시그니처

      • 타입스크립트 인덱스 시그니처는 키로 string 대신 number 타입을 사용할 수 있지만, 명명된 속성은 그 타입을 포괄적인 용도의 string 인덱스 시그니처 타입으로 할당할 수 있어야 한다. ⇒ [i: number]: string

         1interface MoreNarrowNumbers {
 2    [i: number]: string; **// OK**
 3    [i: string]: string | undefined;
 4}
 5
 6const mixesNumbersAndStrings: MoreNarrowNumbers = {
 7    0: '',
 8    key1: '',
 9    key2: undefined,
10};
11
12interface MoreNarrowStrings {
13    **[i: number]: string | undefined; // ERROR: 'number' index type 'string | undefined' is not assignable to 'string' index type 'string'.**
14    [i: string]: string;
15}

인터페이스 병합

  • 두 개의 인터페이스가 동일한 이름으로 동일한 스코프에 선언된 경우, 선언된 모든 필드를 포함하는 더 큰 인터페이스가 추가된다.
 1// 오류 발생 -> 병합 불가
 2interface MergedProperties {
 3    same: (input: boolean) => string;
 4    different: (input: string) => string;
 5}
 6
 7interface MergedProperties {
 8    same: (input: boolean) => string;
 9    different: (input: number) => string; // ERROR: 'different' was also declared here.
10}
11
12// 병합 가능 케이스 : 동일한 이름과 다른 시그니처를 가진 메서드는 정의할 수 있다.
13// 이렇게 하면 메서드에 대한 함수 오버로드가 발생한다.
14interface MergedMethods {
15    **different(input: string): string;**
16}
17
18interface MergedMethods {
19    **different(input: number): string;**
20}
